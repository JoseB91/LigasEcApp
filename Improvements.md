Findings
--------

- LigaPro Serie B still goes through the normal remote flow, so losing RapidAPI access ends up throwing an error alert on the team grid instead of telling the user the data is unavailable. `LeagueRepository` always publishes both leagues as live (`LigasEcApp/Core/Repositories/LeagueRepository.swift:23-41`), `TeamRepository` blindly attempts to hit RapidAPI whenever `league.dataSource == .transferMarket` (`LigasEcApp/Core/Repositories/TeamRepository.swift:25-78`), and `TeamView` has only loading vs list states (`LigasEcApp/LigasEcUI/Team/TeamView.swift:21-55`). Introduce a new league status or data-source case (e.g. `.comingSoon`) so the repository can short-circuit with a typed error and the UI can show the “Coming soon” copy instead of a spinner/alert.
- Cached data can never be refreshed and is not scoped per entity. The Core Data store refuses to overwrite an existing league cache (`LigasEcApp/LigasEcCache/Infra/CoreData/CoreDataLigasEcStore+LeagueStore.swift:20-27`), and the in-memory fallback keeps only one list of teams/players without keying by league/team (`LigasEcApp/LigasEcCache/Infra/InMemory/InMemoryStore.swift:24-52`). Team and player caches also lack timestamps or any `CachePolicy` usage (`LigasEcApp/LigasEcCache/Team/LocalTeamLoader.swift:31-37`, `LigasEcApp/LigasEcCache/Player/LocalPlayerLoader.swift:31-37`).
- Provider configuration is hard-coded and brittle. API hosts, locale, `tournamentStageId` and RapidAPI domains live in code (`LigasEcApp/Core/Composer.swift:29-56`, `LigasEcApp/Core/Repositories/TeamRepository.swift:30-35,60-61`), and the HTTP client happily sends requests with an empty API key if retrieval fails (`LigasEcApp/Core/Composer.swift:59-82`, `LigasEcApp/LigasEcAPI/Infra/URLSessionHTTPClient.swift:19-28`). Move these values into `Config.xcconfig` or a plist so LigaPro season/stage IDs can be updated without a build, surface “API key missing” as a blocking error, and consider keeping an alternate non-RapidAPI provider for Serie B. The same repository logic is duplicated for every data source: both team and player repositories repeat the “try local → fetch remote → save → return” flow with only endpoint/mappers changing (`LigasEcApp/Core/Repositories/TeamRepository.swift:25-78`, `LigasEcApp/Core/Repositories/PlayerRepository.swift:25-79`). Extract reusable loaders (e.g., `RemoteTeamLoader` protocol with strategies per provider) so onboarding a new provider for Serie B or replacing RapidAPI means wiring a new strategy rather than copying two large switch blocks.
- SwiftUI view models are not retained correctly. `TeamView` and `PlayerView` store their reference-type view models in `@State` (`LigasEcApp/LigasEcUI/Team/TeamView.swift:10-55`, `LigasEcApp/LigasEcUI/Player/PlayerView.swift:10-95`), so SwiftUI recreates them whenever the view hierarchy refreshes, re-triggering `.task` network calls and losing in-flight state. Promote them to `ObservableObject` + `@StateObject` and expose `loadIfNeeded()` (similar to `LeagueViewModel` at `LigasEcApp/LigasEcUI/League/LeagueViewModel.swift:33-39`) to ensure each screen only loads once unless the user explicitly pulls to refresh.
- League error handling is incomplete: `LeagueViewModel` has an `errorMessage` property but the view never binds it, so a cache failure silently falls back to hard-coded leagues. Player grouping also relies on Spanish strings, so once you localize the raw strings or switch providers, dictionary lookups will break. Emit a `Player.Position` enum from the mapper and let the UI derive localized labels from that enum instead of literal `"Portero"` keys.

Next steps
----------

1. Decide how you want to flag disabled leagues (new enum case, feature flag, remote config) so the Team screen can show “Coming soon” immediately for Serie B.
2. Refactor the cache stores to support updates + expiration, then add instrumentation/logging so RapidAPI outages are caught before reaching users.
3. Move provider configuration & API key management out of the Composer, enforce a hard failure when the key/provider is unavailable, and abstract provider-specific details so onboarding a new Serie B source is mostly configuration.
4. Clean up the SwiftUI state management for `TeamView`/`PlayerView` and wire `LeagueViewModel` errors into the UI, then add UI tests (per the TODO in `LigasEcApp.swift`) that navigate across leagues/teams to guard the new behavior.
